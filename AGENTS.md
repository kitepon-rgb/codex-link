# AGENTS.md

このリポジトリで作業する Codex や、その他のコーディングエージェント向けの作業指針です。

## プロジェクトの目的

Codex Link は、iPhone から、ユーザー自身の Mac や PC 上で動いている Codex を操作するためのシステムです。

主な構成要素は3つです。

- `apps/ios`: ネイティブ iPhone アプリ。
- `apps/mac-host`: ローカル Codex 連携を担当する macOS Host アプリ。
- `services/relay`: 認証、Host 登録、オンライン状態、ルーティングを担当するマルチテナント Relay。

補助パッケージは、今後以下に置く想定です。

- `packages/protocol`: iPhone <-> Relay <-> Host の共有プロトコル型。
- `packages/codex-client`: Host 側でローカル Codex CLI / app-server と連携するためのコード。

現時点では、このリポジトリはドキュメント中心です。実装を追加するときは、`docs/` に書かれている設計を崩さないでください。

## プロダクトの基本ルール

- Relay を中央 Codex 実行サーバにしない。
- Codex は、選択された Host 上でローカルに動かす。
- Host が、ローカルファイルシステム、ローカルツール、承認、実行結果の副作用を管理する。
- iPhone アプリは SSH クライアントにしない。
- iPhone アプリからプロジェクトフォルダを直接読まない。
- iPhone アプリから raw Codex app-server JSON-RPC を直接話さない。
- Relay はグローバルな Host 一覧を返してはいけない。
- Host に対するすべてのルーティングで、現在のユーザーがその Host にアクセス可能か確認する。
- 実装していない限り、エンドツーエンドプライバシーを実現していると主張しない。
- 永続的なコード状態は GitHub を通す。つまり、ブランチ、コミット、PR、Issue、ドキュメントを使う。

## セキュリティ方針

Relay は、自宅 LAN 上で動かす場合でも、公開されたマルチテナントサービスとして扱ってください。

必要な概念は以下です。

- User
- Device
- Host
- HostAccess
- Connection

単一の共有 API トークンは避けてください。デバイスごとの登録、取り消し可能な認証情報、短命セッション、Host 所有権チェック、ルートごとの認可チェックを優先します。

Relay が保存してよいもの:

- ユーザーやデバイスのメタデータ
- Host のメタデータ
- Host のオンライン状態
- アクセス制御情報
- 最小限の監査メタデータ
- 再接続用の短いイベント投影データ

Relay が保存してはいけないもの:

- プロジェクトフォルダ
- SSH 認証情報
- ローカル Codex の認証状態
- ローカルファイルシステムの内容
- Codex セッションの正本
- 明示的に設計していない長い raw ログ

## Codex 連携

Codex 連携に関する判断は、公式ドキュメントか、ローカルで再現できるスモークテストで確認してください。

名前が似ているだけで、Codex の複数機能が同じものだと推測しないでください。Codex 連携の方式を追加するときは、`rag/third-party/openai-codex.md` に確認状況を書いてください。

最初に採用する Host 連携方式は、実装を広げる前に意図を持って選び、記録してください。

## 開発方針

- 憶測で大きなフレームワークコードを作らず、小さく明示的なモジュールを優先する。
- iPhone、Relay、Host、ローカル Codex の境界をはっきり保つ。
- コードができたら、共有の通信型は `packages/protocol` に置く。
- Host 専用の Codex 詳細を iPhone アプリに持ち込まない。
- Relay の認可処理は、各ルーティング処理の近くに置く。
- 新しい設計判断を追加したら、関連ドキュメントも更新する。
- 機能実装のついでに、関係ないドキュメントを大きく書き換えない。

## フロントエンドとアプリの方針

このプロダクトは、開発者向けの操作盤として落ち着いて使えるものにします。マーケティングサイトのような見た目を優先しないでください。

- まず実際に使えるアプリ画面を作る。
- Host 選択、プロジェクト選択、実行状態、 transcript、承認、Live Activities を優先する。
- UI は情報密度があり、落ち着いていて、見通しがよいものにする。
- 状態や操作を邪魔する装飾を避ける。
- macOS / iOS アプリでは、明確な理由がない限りネイティブの作法に従う。

## 想定するローカル開発ツール

このリポジトリでは、まだ package manager、build、lint、test コマンドを定義していません。

実装が始まったら、採用したコマンドを該当 README に書き、このファイルも更新してください。それまでは以下を守ってください。

- 最終報告で、存在しないテストコマンドを実行したことにしない。
- まだプロジェクトが存在しないためテストできない場合は、そのまま正直に書く。
- TypeScript monorepo を採用する場合は、`pnpm` を優先する。
- iOS / macOS ターゲットでは、Xcode / SwiftPM の標準的な流れを優先する。

## MVP の優先順

以下の順番で、まず一周動く流れを作ります。

1. リポジトリ構成。
2. Relay の骨組み。
3. Mac Host MVP。
4. iPhone MVP。
5. Live Activity MVP。
6. イベント中心の UX。
7. マルチユーザー向けの堅牢化。

MVP の流れが証明される前に、Windows Host、完全な thread/session 互換、完全なエンドツーエンド暗号化、App Store 公開対応へ広げすぎないでください。
